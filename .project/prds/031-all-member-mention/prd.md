# @all Member Mention

## Description

Fircle already supports structured member mentions in post captions and comments, and those direct mentions create notification records for claimed recipients. What it does not support is a family-wide mention primitive for moments when an author wants to notify everyone participating in the family space without manually selecting each person.

This PRD introduces a first-class `@all` mention for posts and comments, but restricts authoring of that broadcast mention to OWNER and ADMIN family members. When an eligible author uses `@all`, Fircle should keep the literal `@all` token in the published content, treat it as a special mention entity rather than plain text, and create mention notifications for all claimed members in the family except the actor. The implementation must preserve the existing structured-mention model, notification preferences, and push behavior already established for ordinary mentions.

In scope for this PRD:
- `@all` authoring support in post captions for OWNER and ADMIN members only.
- `@all` authoring support in top-level comments, replies, and comment edits for OWNER and ADMIN members only.
- Persistence of `@all` as structured mention metadata rather than free-text-only behavior.
- Mention-notification fan-out from `@all` to all claimed family members except the author.
- Deduplication when `@all` and direct mentions target overlapping claimed members.
- Rendering published `@all` tokens consistently in feed and post-detail comment surfaces.

Out of scope for this PRD:
- A new dedicated notification event type for `@all`.
- Expansion of `@all` into rewritten authored text such as explicit member-name lists.
- Role-scoped variants such as `@admins` or `@owners`.
- Post-edit mention authoring if a post-edit flow is introduced later.
- Mentions in per-media captions or unrelated text-entry surfaces.

### Design Decisions

- **Literal token preservation**: Published content should continue showing the exact `@all` token instead of mutating the author's text into multiple explicit member mentions.
- **Structured special mention, not regex-only behavior**: `@all` should be represented in the same overall mention system as normal mentions so rendering, validation, and notifications remain deterministic.
- **Role-gated broadcast capability**: Only OWNER and ADMIN members may author `@all`; regular MEMBER users must not see or successfully submit this special mention.
- **Claimed-members-only delivery**: Notification fan-out should only include claimed members, matching the existing notification contract for mentions and tags.
- **Actor exclusion**: Authors should never receive a notification generated by their own `@all` mention.
- **Recipient deduplication**: If a member is directly mentioned and also included via `@all`, they should receive a single mention notification for that source event.
- **Existing mention event types remain authoritative**: `@all` notifications should continue using `POST_MENTION_CREATED` and `COMMENT_MENTION_CREATED` so current notification preferences and push filtering continue to apply.
- **Single-family boundary enforcement**: `@all` only targets members inside the active family and must not rely on any future multi-family/global user concept.
- **One special token per entity**: The server should reject duplicate `@all` mention entries within the same post/comment payload because repeated tokens do not add meaning and complicate persistence.

### User Stories

- **As an** owner or admin posting an important update, **I want** to mention `@all`, **so that** everyone in my family who has claimed their profile is notified without manual selection.
- **As an** owner or admin writing a comment, **I want** to use `@all` in the same mention flow as direct member mentions, **so that** group discussions can notify everyone consistently.
- **As a** regular family member, **I do not want** access to broadcast-style mentions, **so that** high-visibility notifications remain limited to trusted family coordinators.
- **As a** family member reading the feed or comment thread, **I want** published `@all` content to remain visibly written as `@all`, **so that** the author's intent is clear at a glance.
- **As a** claimed family member, **I want** `@all` notifications to obey my existing mention notification preferences, **so that** I do not need a separate settings model just for this feature.
- **As a** maintainer, **I want** `@all` to reuse the existing structured mention and notification foundations, **so that** the implementation stays consistent and avoids special-case drift across read and write paths.

**Goal:** Extend the existing mention contracts so post/comment authoring, persistence, and rendering can represent a special `@all` token cleanly.

#### Tasks

- [x] Update the mention draft/client helper model in [src/components/feed/mention-helpers.ts](src/components/feed/mention-helpers.ts) so mention entries can represent either a concrete family member or the special `@all` token.
- [x] Introduce an explicit special-mention identifier/kind contract in server input handling within [src/server/api/routers/post.ts](src/server/api/routers/post.ts) instead of overloading `memberId` with an unsafe fake value.
- [x] Evaluate and implement the minimal Prisma schema evolution in [prisma/schema.prisma](prisma/schema.prisma) needed to persist `@all` mention ranges alongside current mention records while preserving existing member mention behavior.

## Acceptance Criteria

- [ ] Typing `@` in post captions and comment inputs can surface `@all` as a suggestion alongside direct member mentions for OWNER and ADMIN users only.
- [ ] OWNER and ADMIN users can insert `@all` using the same keyboard and pointer interactions used for normal mentions.
- [ ] Regular MEMBER users do not see `@all` in mention suggestions and cannot successfully submit it through the API.
- [ ] Published post and comment content keeps the literal `@all` token visible instead of rewriting authored text.
- [ ] `@all` is persisted as structured mention metadata, not as plain text requiring regex-only interpretation.
- [ ] A post containing `@all` creates mention notifications for every claimed family member except the author.
- [ ] A comment or reply containing `@all` creates mention notifications for every claimed family member except the author.
- [ ] Editing a comment to include `@all` updates persisted mention metadata and creates the expected mention notifications without duplicate fan-out.
- [ ] Members who are directly mentioned and also covered by `@all` receive only one mention notification for the same source event.
- [ ] Unclaimed family members do not receive `@all` notifications.
- [ ] Existing mention notification settings and push preference behavior continue to govern `@all` notifications with no new preference surface required.
- [ ] Existing direct member mentions in posts and comments continue to work with no regression in rendering, linking, or notification delivery.
- [ ] Targeted tests for helper logic, router behavior, and notification fan-out pass.

## Further Considerations

- A follow-up PRD can introduce role- or audience-scoped mention groups such as `@admins` only after `@all` proves the special-mention model is stable.
- If analytics or moderation later need to distinguish direct mentions from broadcast mentions more explicitly, the persisted special-mention kind introduced here should be reused rather than inferred from notification volume.
- If a future post-edit flow is added, it should adopt the same `@all` authoring, validation, and deduplication behavior defined in this PRD instead of creating a parallel implementation.
